# Phase 1 — Foundation

## Problem Statement

Slate needs a working foundation: a data model, a file-based store, a programmatic library API, and a CLI. Without this, no agent or human can use Slate to track tasks and PRDs.

## Solution

Build a minimal but complete issue tracking system with:

- A single `IStore` interface and `LocalFileStore` implementation that persists PRDs and tasks as Markdown files with YAML frontmatter.
- A `Slate` library class that exposes the store through a clean, programmatic API.
- A CLI built on commander that exposes the same operations via `slate prd` and `slate task` subcommands, with body content passed via stdin to avoid shell escaping issues.
- Dependency resolution: tasks can depend on other tasks, and a `resolve` command marks a task done and detects unblocked dependents.

## User Stories

1. As an agent, I want to create a PRD via the CLI, so that I can define a feature initiative with a title and have it persisted as a file.
2. As an agent, I want to create a task via the CLI with a title, priority, optional PRD binding, and body content piped via stdin, so that I can capture structured work items without worrying about shell escaping.
3. As an agent, I want to list all tasks filtered by status, so that I can find tasks matching a specific state (todo, in-progress, done, blocked).
4. As an agent, I want to update a task's status, so that I can progress work through the lifecycle (todo → in-progress → done).
5. As an agent, I want to list all PRDs, so that I can see what initiatives exist in the project.
6. As an agent, I want to view a PRD's details, so that I can review its metadata.
7. As an agent, I want to mark a task as done and automatically detect which dependent tasks are now unblocked, so that I can efficiently progress work through dependency chains.
8. As an agent, I want the system to detect and reject dependency cycles, so that invalid task graphs are caught early.
9. As a library consumer, I want to instantiate `Slate` with a store directory and use `prds.create()`, `tasks.create()`, `tasks.query()`, `tasks.resolve()`, and other methods programmatically, so that I can integrate Slate into agentic harness extensions.
10. As an agent, I want all operations to return `Result<T, E>` with discriminated error unions, so that I can handle errors explicitly without exceptions.
11. As an agent, I want each PRD and task to be stored as a separate Markdown file with YAML frontmatter, so that git diffs are clean and parallel editing produces no cross-entity conflicts.
12. As a developer, I want the store directory structure to be `slate/prds/` for PRDs and `slate/tasks/` for tasks, so that the physical layout is predictable and consistent.
13. As an agent, I want to create tasks that are not bound to any PRD (ad-hoc tasks), so that I can capture bugs, maintenance items, and other work that doesn't fit a structured initiative.
14. As a user, I want the CLI to auto-generate sequential IDs (e.g., `task-001`, `prd-001`) for new entities, so that I don't need to manage ID uniqueness manually.
15. As a user, I want timestamps (created, updated) to be in ISO 8601 format and automatically set on creation and updated on modification, so that entity lifecycle is tracked without manual input.
16. As a developer, I want the `Slate` class to wire `LocalFileStore` to `IStore` internally, so that consumers have a single entry point and don't need to manage store instantiation.
17. As a developer, I want modules to depend on interfaces, not implementations, so that tests can inject mocks and future storage backends can be swapped without touching consumer code.
18. As a tester, I want integration tests that verify behavior through the public interface (CLI commands and library API), so that the tests validate what the code does, not how it does it.

## Implementation Decisions

- **Single `IStore` interface**: One interface handles both PRD and task operations. A single `LocalFileStore` implementation provides the concrete file-based storage. This avoids premature splitting into separate PRD and task store modules — both entity types share the same storage pattern.
- **Markdown with YAML frontmatter**: Each entity is a `.md` file. Structured fields (id, title, status, priority, dependencies, prd, timestamps) live in YAML frontmatter. Freeform body lives in the Markdown body below the frontmatter.
- **One file per entity**: Each PRD and task gets its own file in `slate/prds/` or `slate/tasks/`. This ensures merge safety, clean diffs, and parallel editing.
- **CLI built on commander**: Commander handles argument parsing, option types, help generation, and subcommand dispatch. Body content is passed via stdin to avoid shell escaping issues.
- **Optional PRD binding**: Tasks can reference a PRD by ID, but don't have to. This keeps the model flexible for structured work and ad-hoc tasks.
- **Sequential ID generation**: IDs are auto-generated as `task-NNN` and `prd-NNN` based on the highest existing number plus one.
- **Status values**: `todo`, `in-progress`, `done`, `blocked`.
- **Priority values**: `high`, `medium`, `low`.
- **Dependency cycles are invalid**: The system must detect and reject cycles when setting dependencies or resolving tasks.
- **Scan-first approach**: Querying tasks reads all entity files from the store. This is O(n) but fast enough for the expected scale (20–100 tasks).
- **No build step**: The project runs source directly via bun. No compilation or bundling.
- **Result type**: All fallible operations return `Result<T, E>` with discriminated error unions. No throwing exceptions.

## Testing Decisions

- **Integration tests first**: Tests exercise the full stack from a user-facing point of view — CLI commands and library API. Tests verify file content on disk after CLI operations, and verify library return values after programmatic operations.
- **Test the store via the CLI**: After running a CLI command, verify the resulting file exists in the correct directory with correct YAML frontmatter and body content. This tests the full path: CLI → commander → command handler → store → file.
- **Test the library via direct usage**: Instantiate `Slate`, call its methods, and verify return values. Tests the full path: library → store → file.
- **Never test internals**: Don't test private functions, implementation details, or helper methods in isolation. Test what the code does, not how it does it.
- **Modules to test**:
  - `IStore` / `LocalFileStore` — tested through CLI commands (prd create, task create, task update) and library usage.
  - `Slate` class — tested through programmatic API usage.
  - Dependency resolution — tested through `task resolve` CLI command and library method, verifying file content before and after.
- **Test environment**: Tests run in a temporary directory to avoid polluting the project's actual store.

## Out of Scope

- Remote hosting or sync
- User accounts or permissions
- Time tracking or estimates
- Epics, milestones, or other hierarchy levels
- Automation or triggers
- TUI (terminal UI)
- Extension packaging and discovery
- `slate/index.json` fast-lookup index
- `slate/config.json` project-level configuration
- SQLite or other non-file storage backends

## Further Notes

- The `result.ts` utility already exists in `src/utils/result.ts` — no need to build it.
- Phase 1 is the foundation. Phase 2 (agentic integration) and Phase 3 (polish) build on top of this.
- The CLI should support both `--title` and `--priority` as flags, with body via stdin. Body is optional.
- The `Slate` class is the public entry point. It wires `LocalFileStore` to `IStore` internally.
